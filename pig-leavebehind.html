One leak, three rooms

Private search as the application layer on top of anonymous broadcast. The shared last-mile IP leak, the two layers I have built, and the four open seams between the Flashbots transport and a usable system.

dmarz, with Claude · June 2026 · companion to the talk; all numbers computed or measured locally

1. The thesis

You can win at the application layer and still lose at the network layer.

I am not building private inference. For search the content of a query is rarely the sensitive thing. The fact that I asked it is. So this is an unlinkability story, not a confidentiality one.

A network observer or the receiving endpoint learns source IP, destination, timing, volume, and packet shape, bound to a stable identity, regardless of payload encryption. TLS protects content, not shape. The application-layer primitive in each use case hides a different thing and leaves this one network-layer leak untouched. That orthogonality is the whole argument, and Flashbots already named it: "even when infrastructure runs in TEEs, operators are still able to learn metadata opening up surface area for selective censorship and adversarial MEV strategies." [1]

2. Three rooms, one leak

Room 1 · BuilderNet

An orderflow provider sends bundles to a builder instance. Orderflow is encrypted to the TEE, so content is safe. The connection is plain mTLS, and the operator at the endpoint learns sender IP, timing, and volume: selective censorship by network identity, latency discrimination, traffic-blocking of a known counterparty. This is the partial case: attestation and the attested-peer orderflow-proxy [2] blunt some of it, which is exactly why it is the easy leg and the next two are the hard ones. The proxy authenticates and signs; it does not anonymize the sender.

Room 2 · the last mile to a TEE

A journalist or researcher queries an attested inference enclave, where the query pattern, not the content, is sensitive; or an agent whose query volume fingerprints its operator. The TEE protects prompt content and does nothing for the last mile; the connection still carries the user's IP. Secure inference is not anonymous inference. Confidential Prompting [3] secures prompt content in a CVM and is silent on client IP, which is the gap, not a claim it makes.

Room 3 · search

Against a normal engine you leak IP and raw query. Against an oblivious (PIR) engine the content is hidden but you still open a connection that leaks IP, timing, and query count. Credit Wally [5], not Tiptoe, for stacking PIR on an anonymous transport: Tiptoe states it "does not hide when a client makes a query, or how many queries the client makes" [4]; Wally assumes an anonymous network plus fake queries (and relaxes to differential privacy, not PIR). That two-layer stack, PIR for content over an anonymous transport for origin, is mine.

The escape that fails: a residential proxy does not anonymize you, it relocates the all-seeing observer to a commercial party bound to your billing identity (see the 911 S5 takedown, over 19M IPs across 190 countries [6]). Real anonymity is unlinkability split across non-colluding parties.

3. The pipe, and what it forces

Real anonymity needs anonymous broadcast, and Flashbots already built the pipe. I do not rebuild it. I sit on top of it.

THE ANONYMITY TRILEMMA (buy anonymity with latency OR bandwidth: a frontier, not pick-2-of-3) Tor / onion Nym / mixnet DC-net (Chaum 1988) weak anon strong anon strong anon, the live branch | ZipNet (eprint 2024/1227) | ADCNet (flashbots/adcnet) | Flashnet (auction scheduling) | NAMP <- transport (Flashbots) ---------------------------------------------------------------------------- MY LAYERS (on top): access: reputation-gated egress Semaphore + gateway-counter nullifier [BUILT+LIVE] application: Periscope oblivious search (LWE) [BUILT] FOUR SEAMS = THE RESEARCH: A return path B payment C issuance cost D portability vs linkage (FLAGSHIP)

Two facts kept separate: anonymity = anytrust (one honest server hides everyone); liveness = the separate honest-majority axis. TEEs buy liveness and bandwidth, not anonymity. ZipNet, verbatim: "anonymity does not rely on the security of TEEs." NAMP staking integration is my own extrapolation, not in the writeup.

The trilemma is a lower-bound frontier, not a clean pick-two-of-three [7]: against a global passive adversary you buy strong anonymity by spending added latency or added bandwidth. The Flashbots line spends bandwidth to keep latency low.

What you inherit the moment you turn it on: spam. Anonymity removes the one free rate-limit the internet gave you, the IP. This is transport-independent, true for Tor and DC-net alike. So the first thing on top of the pipe is not the application. It is an access layer.

4. What is built

Reputation-gated egress (the access layer), deployed 2026-06-08, p50 1.9s against real Google. A client proves zk membership in a Semaphore v4 set, dials a Tor onion service with no exit node, and the gateway tunnels out from one clean IP. Home IP and Tor exit never touch the target. Anti-spam is a per-epoch nullifier rate-limited by a gateway-side budget counter, not in-circuit Shamir reconstruction; the gateway counts, and full RLN slashing is designed, not built. The gateway is trusted for availability and honest-but-curious for unlinkability: it sees the nullifier and outbound timing, not the home IP. The live anonymity set is 1 leaf today, a proof of mechanism that grows as members join; the numbers live in the weak-anonymity, no-cover-traffic corner (Tor) by design, and ride Flashnet later.

Periscope (the application): oblivious search, Tiptoe/SimplePIR style, secret-key Regev LWE (n=1024, q=2³², σ=6.4). The client embeds and encrypts locally; the server ranks over ciphertext and never sees the query or which record. Per-query bandwidth is ~29 KiB up / ~8 KiB down at this corpus size (6143 chunks), scaling ~√N; 20/20 correctness, 49ms median, live on 3 nodes at ~$18/mo. Raw SimplePIR is 242 KB/query at a 1 GB database [8]; Periscope is smaller only because the index is smaller. Two honest error sources: 0.56 top-10 overlap (quantization, on the built thing) and 0.81 router accuracy (~19% of queries hit the wrong shard). The TEE belongs in the build plane, attesting honest index construction; confidentiality is LWE, not hardware.

5. The four seams (research questions)

The leak is shared across all three rooms; the fix surface is not. Two seams the mempool use case hid because mempool broadcast is fire-and-forget. Two it shares but never centered because they are transport-independent. On each I name the prior art up front and scope the open part narrowly.

Seam A · the broadcast return path

mempool HID it

Search and inference are request/response, so I must get an answer back to a sender with no address. The general async return path is a private mailbox: authorized DPF-write to deliver (Express [10], Sabre) and PIR-read to collect (Talek [11]). That is known. The store-and-handle trick is the synchronous special case; the fetch is itself a PIR. Open: (a) can the PIR-read be sharded or centroid-routed the way Periscope shards its index, so a client scans only its shard, not a linear scan over all mailboxes at search scale; (b) does folding the collect-read into Flashnet's IBLT auction as a second message class leak via its different size and shape, given the allocator currently models one class.

Seam B · anonymous payment without rebuilding the graph

mempool HID it

RLN gives anonymous metering, not anonymous payment. Carrying value in one anonymous credential is solved: ACT [12] is a KVAC/BBS credential carrying a balance, spent via range proof, revealing a fresh nullifier and returning signed anonymous change; Nym zk-nyms [13] are a deployed cousin. I adopt it, I am not reinventing a coin. Open, in my setting: (1) one gateway-issued KVAC running dual-mode (an ARC [14] presentation limit for the free tier, an ACT balance for paid egress) on one nullifier store without the two tiers re-linking; (2) collapsing the funding event, where money meets identity, to one padded batched top-up so the residual leak is a single infrequent event; (3) binding the spend nullifier, the RLN nullifier, and the return-path reply credential to one query without re-linking all three. The double-spend check is online and must be atomic with insertion.

Seam C · issuance cost as the Sybil root

mempool SHARES it, transport-independent

RLN is a meter, not a mint: it bounds throughput per credential, not the number of credentials. Let C(N) be the cost for one adversary to control N valid leaves. The known frontier: pure stake is linear and splittable; proof-of-personhood caps near one per human (World ID already uses Semaphore); pluralistic composition is the realistic path toward Vitalik's N² ideal [15]. Open: find an issuance mechanism whose C(N) is at least linear and ideally superlinear, cheap at N=1, and anonymous at issuance (the issuer learns only "one new member"). Can RLN's slashable stake double as the issuance cost, or does the stake-splitting linearity result [16] force a personhood gate on top? Does anonymous issuance preclude social-graph methods, since those read graph position?

Seam D · portability vs linkage  ★ flagship

mempool SHARES it, transport-independent

My RLN credentials are spendable across egress points. Good for the user. But two egresses in the same epoch see the same nullifier for the same user, which is exactly what lets each rate-limit and exactly what correlates them. Privacy Pass RFC 9576 [17] names the linkage hazard; ARC bounds cross-context use; neither resolves the tension. Falsifiable conjecture: for a user spending one credential at egress points {E₁..Eₖ} in epoch t, a global aggregate rate limit across all Eᵢ and unlinkability against any coalition of up-to-threshold egresses are mutually exclusive under any shared-nullifier construction, because the object that lets two egresses sum your usage is the object that lets them correlate you. Open: does a re-randomizable token-bucket carried in an ARC-style credential, with the global counter reconciled out of band, escape the tradeoff, or is it fundamental? This is the most defensible "no one has solved this" claim I have.

There are four seams, matching the talk. "Where does the work belong" is positioning, not a question. A genuine fifth candidate, if wanted: the agent-era query-cost blowup. A human types one short query; an agent expands it into a fan-out of long, multi-vector sub-queries, each a full PIR pass forcing its own return delivery. Tiptoe and Periscope economics assume the human; the cost model for the agent is unstudied, and it reconnects seam A's scheduling to the PIR.
